# AGENTS.md

## Project
Local, fully offline app that uploads PDF books in any language and translates them entirely into a target language of the user's choice (default Vietnamese) using local LLMs via Ollama. No cloud calls anywhere. Personal single-user tool — no auth, no multi-tenancy.

The first working version is complete and verified end-to-end; the sections below record the architecture and conventions to follow when extending it.

## Stack
- **Backend**: Python 3.11 + FastAPI, in `backend/`. Async throughout (calls to Ollama are HTTP).
- **Frontend**: Angular 21, standalone components + signals (no NgModules, no NgRx), in `frontend/`.
- **Local LLMs via Ollama** (`localhost:11434`, must already be running):
  - `translategemma:latest` — rough/fast translation pass.
  - `qwen2.5:14b-instruct-q4_K_M` — literary polish pass (fallback: `qwen2.5:7b-instruct-q4_K_M` if too slow on this machine).
- **PDF parsing**: PyMuPDF (`fitz`), using `page.get_text("dict")` for font-size metadata to distinguish headings from paragraphs.
- **Persistence**: SQLite (stdlib `sqlite3`, no ORM) for book/job status rows; per-book JSON files (`structure.json`, `glossary.json`) for content. No Postgres, no SQLAlchemy.
- **Job execution**: in-process `asyncio.create_task`, sequential (not concurrent) chunk processing — no Celery/Redis. Translation runs as **two whole-book passes** (`rough_translating` then `polishing`), not interleaved per chunk — each pass loads only its own model for the whole book, so low-RAM machines that can't keep both models warm don't pay a reload cost every chunk. Glossary is fully built by the end of the rough pass, before polishing starts (block model has separate `rough_text` and `translated_text` fields — see `book_structure.py`).
- **Source/target language are free text, not a fixed enum**: `books.target_lang` is a plain `TEXT` column (e.g. `"Tiếng Nhật"`, `"German"`) chosen per book at upload time via a curated `<select>` dropdown + "Khác…" free-text fallback in the UI (default `"Tiếng Việt"`) — not restricted to a hardcoded `en`/`fr` list. All prompts in `translate.py`/`summarize.py`/`rag.py` are parameterized on source/target language strings and written in English for the meta-instructions (robust across arbitrary target languages), with the actual translated/summarized/chat content produced in `{target_lang}`. Verified live end-to-end (translate + auto-index + chat + summarize) with English→Japanese and German→Vietnamese, not just the original English/French→Vietnamese path. **Known limitation**: `translate.py`'s proper-noun/glossary-consistency heuristic (`MID_SENTENCE_CAPITALIZED`) relies on Latin upper/lowercase distinction, so it silently finds zero candidates (no error) for source languages without case (CJK, Arabic, Thai, ...) — glossary consistency just doesn't kick in there, translation quality otherwise unaffected.
- **Source language is auto-detected, not user-entered**: `books.source_lang` starts empty at upload (`upload_book` no longer takes a `source_lang` field at all) and is filled in once, right after PDF extraction, by `translate.detect_source_language()` — a single rough-model call over the first ~1000 chars of extracted text, asked to name the language in Vietnamese (e.g. `"Tiếng Đức"`). Cached on both `BookStructure.source_lang` and the `books` row (`job_runner._get_or_detect_source_lang`) so retry/re-translate never re-detects. UI shows "Đang nhận diện…" for the empty-string pending state. Rationale: users can't reliably type a language name up front for a PDF whose language they may not even be sure of, and the extracted text is sitting right there already.
- **qwen2.5 (polish model) has a reproducible "thinks out loud in Chinese" failure mode on literary/narrative prose** — found via live testing while verifying the language generalization above, unrelated to it (reproduces on plain English→Vietnamese too, confirmed 3/3 broken on narrative sentences like "the old lighthouse..." vs. 0/3 on technical content like the HTML5 Canvas book). Not something prompt tweaking reliably fixes — tried explicit "don't explain your reasoning" instructions and lower temperature (0.5→0.3), neither eliminated it. **Fix**: `translate.polish()` now detects unexpected CJK characters in its own output (`_has_unexpected_cjk_leak`, skipped when the target language itself is Chinese/Japanese/Korean via `_target_expects_cjk`) and retries up to `POLISH_MAX_ATTEMPTS` (3) times; if still leaking after all attempts, falls back to returning the (already clean) rough-pass text unpolished rather than shipping garbled output — same fallback-chain idiom as `_split_or_fallback`. Verified live: a sentence that leaked CJK on 15/15 raw attempts still produced a 100% clean final result for the user every time, at the cost of skipping the literary polish for that one paragraph.
- **Output**: backend renders HTML from `structure.json` via Jinja2; served as a fragment to Angular (`[innerHTML]`) and also saved as a standalone `output.html` file per book. The reader uses `DomSanitizer.bypassSecurityTrustHtml()`, **not** `.sanitize()` — Angular's sanitizer strips `id`/`data-*` attributes the backend relies on (chapter anchors, page markers). This is safe because the HTML is entirely backend-generated with all extracted text escaped (`markupsafe.escape()`), never raw third-party input.
- **Styling**: all Angular component styles are `.scss` (not `.css`). Colors are CSS custom properties (`--color-*`) defined once in `frontend/src/styles.scss`'s `:root` block — never hardcode a hex color in a component stylesheet, add/reuse a token instead. `book.html.jinja2` defines the **same token names/values** in its own `:root` block (plain CSS — that file has no build step, so it can't literally share the SCSS source; keep the two in sync by hand when a color changes).
- **Dark mode**: `frontend/src/app/core/theme.service.ts` toggles a `.dark` class on `document.documentElement` (app-wide, not just the reader) and persists to `localStorage` (`theme-dark`); `:root.dark { --color-*: ... }` in `styles.scss` supplies the dark values. `book.html.jinja2` has its own independent vanilla-JS version of the same toggle (same localStorage key by convention, but never actually shared — different origin). **Reader font size** is reader-only: a `--reader-font-scale` CSS variable set inline by `reader-page.ts` (persisted under `reader-font-scale`), applied to `.reader p`'s `font-size` via `calc()` — headings use fixed `rem` sizes and intentionally don't scale.
- **Glossary is internal-only, no user-facing page**: `glossary.json` (built during the rough pass, see above) exists purely so `polish()` translates proper nouns/terms consistently across chunks — there used to be a `/books/:id/glossary` page + `GET`/`PUT /api/books/{id}/glossary` for viewing/editing it, removed on request because it surfaced too many extracted terms to be useful. If a UI for it is wanted again, the extraction/consistency mechanism (`translate.py`'s `extract_glossary_terms`/`new_capitalized_terms`, `job_runner.py`'s `_load_glossary`/`_save_glossary`) never went away — only the display layer did.
- **Chapter summaries** (`/books/:id/summary`): a second job type (`jobs.job_type = 'summarize'`, reuses the same `jobs` table/polling infra as translation). `pipeline/summarize.py` splits blocks into chapters by level-1 heading, summarizes each chapter's *translated* text via the polish model, map-reducing chapters that exceed `SUMMARY_GROUP_TOKEN_BUDGET`. Stored in `data/books/{id}/summaries.json` keyed by heading block id.
- **RAG chat** (`/books/:id/chat`): a third job type (`jobs.job_type = 'index'`) embeds the book's *translated* text via `bge-m3` (`/api/embed`, batched) into small ~500-token retrieval chunks (`pipeline/rag.py`, distinct from the 1800-token translation chunks — coarser chunks dilute retrieval relevance). Stored as `embeddings.npy` + `chunks_index.json` per book, no vector DB — brute-force cosine similarity is instant at this scale (a few hundred to ~1500 chunks). Chat itself streams (`ollama_client.chat_stream()`, `stream: true`) — the one deliberate exception to the rest of the app's poll-a-job convention, since a live Q&A conversation shouldn't feel like it hung for 15-30s. History is a flat JSON file (`data/books/{id}/chat_history.json`), same pattern as `glossary.json`/`summaries.json`. **Chat temperature is pinned to 0.2** (`rag.CHAT_TEMPERATURE`), well below Ollama's qwen2.5 default (~0.7-0.8) — at default temperature, qwen2.5 would occasionally code-switch into Chinese mid-answer, especially on off-topic questions with no retrieved context to anchor generation; confirmed fixed (0/6 code-switches after the change, vs. reproducible before). **Known residual limitation**: even at low temperature and with an explicit "don't use outside knowledge" instruction, qwen2.5:14b still sometimes volunteers a real-world answer to off-topic questions after correctly stating it's not in the book (~1/3 of trials) — a real quantized-model instruction-following limit, not a bug in this codebase's prompt plumbing. Not worth chasing further with more prompt engineering; if this matters later, the fix is a smaller/less "helpful" model or a post-hoc refusal classifier, not another prompt tweak.
- **Chat markdown rendering** (`frontend/src/app/features/chat/chat-markdown.ts`): chat responses are LLM output, not backend-escaped content like the rest of the app — so unlike the reader's `bypassSecurityTrustHtml()` (safe because content is pre-escaped upstream), chat HTML is run through `marked` (markdown → HTML) then **DOMPurify-sanitized** before `bypassSecurityTrustHtml()` is allowed to touch it. Math (`$…$` / `$$…$$`) is extracted into placeholder tokens and rendered via KaTeX (`output: 'html'`, not `htmlAndMathml` — simpler DOMPurify story, no MathML tags to allow) *before* markdown parsing, then swapped back in after, so `marked` can't mangle LaTeX syntax (e.g. underscores read as emphasis). Code blocks are highlighted via `highlight.js` in a custom `marked` renderer. Both `katex.min.css` and a single highlight.js theme (`github-dark`, always dark regardless of the app's own light/dark mode — same convention as `--color-code-bg` elsewhere) are registered as global build-time styles in `angular.json`'s `styles` array, not `styles.scss` — component-scoped SCSS can't style content injected via `[innerHTML]` since Angular's view-encapsulation attributes never get added to it.
- **Angular `@if (signal(); as x)` pitfall**: the condition uses JS truthiness of the signal's *value*, not a null-check — `@if (streamingReply(); as partial)` silently skips the whole block when `streamingReply()` is `''` (a valid "waiting for first token" state), not just when it's `null`. Found live via a streaming chat loading indicator that never rendered. Fix: check for `!== null` explicitly, don't rely on `as` when a legitimate falsy-but-meaningful value (`''`, `0`, `false`) can occur.

## Explicit non-goals (don't add these)
- No `torch`/`transformers`/`ctranslate2`/NLLB — translation only goes through Ollama's REST API.
- No auth, no multi-user support, no cloud deployment.
- No NgRx — plain injectable signal stores only.
- No `resource()`/`httpResource()`/`rxResource()` in Angular yet — verified still `@experimental` in the installed 21.2.17 typings (`node_modules/@angular/*/types/*.d.ts`); revisit when promoted stable. Polling uses the modern stable interop pattern instead: RxJS `timer` + `exhaustMap` + `takeWhile` converted via `toSignal`.
- No iframe for the reader view.

## Running locally
- Backend: `cd backend && source .venv/bin/activate && uvicorn app.main:app --reload --port 8000`
- Frontend: `cd frontend && npm start` (dev server on **`localhost:4210`**, not the Angular default 4200 — this machine runs another unrelated project's dev server on 4200, so the port is pinned in `package.json`'s `start` script and `.claude/launch.json` to avoid clashing with it. `CORS_ORIGINS` in `backend/app/config.py` is set to match; largely moot in dev since `proxy.conf.json` makes API calls same-origin anyway).
- Ollama must be running (`ollama serve`) with both models pulled (`ollama list` to check).

## Working conventions
- Work in small, verifiable steps — implement one module/endpoint at a time, verify it (manually via `/docs`, a script, or the Preview tool), then check in with the user before moving to the next step.
- Prefer stdlib and the minimal dependency set already listed in the plan over adding new libraries.
- Keep prompts to local LLMs and any translated content out of chat responses beyond short illustrative snippets — don't reproduce large verbatim excerpts of source books in conversation.
